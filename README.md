# 1c-maintenance

Комплект bash-скриптов для ежедневного/еженедельного регламентного обслуживания
связки **Astra Linux + PostgreSQL + 1С:Предприятие** (конфигурации Бухгалтерия
и ЗУП, клиент-серверный режим, платформа 1С 8.3.27.2214).

Задачи: логические бэкапы баз PostgreSQL, проверка их целостности и восстановимости,
контроль состояния СУБД (долгие запросы, блокировки, место на диске), контроль
кластера серверов 1С через `rac`, настройка и разбор технологического журнала.

Управление кластером — через `rac`/`ras` с учётной записью администратора кластера.
PITR не используется — бэкапы логические (`pg_dump`), без непрерывного архивирования WAL.

## Структура

```
1c-maintenance/
├── etc/
│   ├── env.conf              # общий конфиг: пути, ИБ↔БД, retention (ЗАПОЛНИТЬ)
│   └── rac.secret.example    # шаблон учётных данных администратора кластера rac
├── bin/
│   ├── common.sh              # общие функции: log(), die(), require_cmd()
│   ├── pg_backup.sh           # ежедневный pg_dump -Fc + ротация + проверка места
│   ├── pg_backup_verify.sh    # проверка целостности свежего дампа (pg_restore --list)
│   ├── pg_health_check.sh     # долгие запросы, блокировки, место под PGDATA
│   ├── pg_maintenance.sh      # еженедельно: ANALYZE + точечный VACUUM по bloat
│   ├── rac_health_check.sh    # состояние кластера 1С через rac (сеансы, процессы)
│   ├── tj_analyze.sh          # разбор технологического журнала за сутки
│   ├── tj_cleanup.sh          # доп. очистка старых файлов техжурнала
│   └── restore_test.sh        # еженедельный тест восстановления бэкапа
├── tj_setup/
│   └── logcfg.xml             # конфиг технологического журнала (EXCP/DBMSSQL/TLOCK/TDEADLOCK)
├── cron/
│   └── 1c-maintenance.cron    # готовое расписание cron
├── logrotate/
│   └── 1c-maintenance.logrotate  # ротация логов самих скриптов
└── Что проверить на сервере Астра Линукс перед установкой скриптов.md
```

## Требования

- Astra Linux (Special/Common Edition)
- PostgreSQL (стандартная сборка из репозитория Astra/PGDG)
- 1С:Предприятие, платформа 8.3.27.2214, клиент-серверный режим
- Утилита `rac` из комплекта поставки платформы 1С
- Учётная запись администратора кластера 1С (логин/пароль для `rac`)
- Права на `pg_dump`/`psql`, включая `CREATE`/`DROP DATABASE` для тестового восстановления

## Установка

Перед установкой — пройтись по чек-листу
[«Что проверить на сервере Астра Линукс перед установкой скриптов»](./Что%20проверить%20на%20сервере%20Астра%20Линукс%20перед%20установкой%20скриптов.md)
(пути PostgreSQL и `rac`, аутентификация, расположение техжурнала, права,
`PATH` для cron).

1. Скопировать каталог `1c-maintenance/` на сервер в `/opt/1c-maintenance`
   (пути в скриптах и cron-файле на это рассчитаны).
2. Заполнить `etc/env.conf`:
   - имена информационных баз 1С и соответствующих им баз PostgreSQL (`INFOBASES`);
   - путь к `rac` (`RAC_BIN`), адрес `ras` (`RAS_HOST`/`RAS_PORT`);
   - каталог и глубину хранения бэкапов (`BACKUP_DIR`, `BACKUP_RETENTION_DAYS`);
   - каталог техжурнала (`TJ_LOG_DIR`), срок хранения (`TJ_RETENTION_DAYS`).
3. Скопировать `etc/rac.secret.example` в `etc/rac.secret`, заполнить реальные
   логин/пароль администратора кластера, выставить `chmod 600` / `chown root:root`.
   **Файл `rac.secret` не должен попадать в git.**
4. Разместить `tj_setup/logcfg.xml` там, где его ожидает кластер серверов 1С
   (расположение зависит от способа запуска `srv1cv8`/`ras` на конкретном сервере —
   проверить перед копированием).
5. Проверить формат вывода `rac process list` на сервере и сверить с разбором
   в `bin/rac_health_check.sh` (имена свойств могут отличаться между сборками платформы).
6. Установить расписание: `crontab -u postgres /opt/1c-maintenance/cron/1c-maintenance.cron`
   (пользователь должен иметь права на PostgreSQL-операции и чтение `rac`).
7. Установить ротацию логов: скопировать `logrotate/1c-maintenance.logrotate`
   в `/etc/logrotate.d/1c-maintenance`.

## Расписание по умолчанию

| Время | Скрипт | Периодичность |
|---|---|---|
| 02:00 | `pg_backup.sh` | ежедневно |
| 02:30 | `pg_backup_verify.sh` | ежедневно |
| 03:00 | `tj_analyze.sh` | ежедневно |
| 03:15 | `tj_cleanup.sh` | ежедневно |
| 08:00, 14:00 | `pg_health_check.sh` | ежедневно |
| 08:05 | `rac_health_check.sh` | ежедневно |
| 01:00 вс | `pg_maintenance.sh` | еженедельно |
| 04:00 вс | `restore_test.sh` | еженедельно |

## Логи

Каждый скрипт пишет в свой файл в `$LOG_DIR` (по умолчанию `/opt/1c-maintenance/logs/`).
Внешних уведомлений (email/Telegram) нет — только файлы логов; ротация — через
`logrotate/1c-maintenance.logrotate`.

## Известные ограничения

- `tj_analyze.sh` — базовый текстовый разбор (grep/awk) технологического журнала,
  не полноценный парсер формата ТЖ.
- `rac_health_check.sh` рассчитан на разбор вывода `rac` построчно ("имя: значение") —
  требует проверки на конкретной сборке платформы перед боевым использованием.
- Основная ротация технологического журнала выполняется самой платформой 1С
  (атрибут `history` в `logcfg.xml`); `tj_cleanup.sh` — дополнительная страховка.
