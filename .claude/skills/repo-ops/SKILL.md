# 1c-maintenance: работа с репозиторием

Проект — набор bash-скриптов для регламентного обслуживания PostgreSQL и кластера
серверов 1С (БП, ЗУП) на Astra Linux. Это не сервис с постоянным процессом, а
набор одноразовых cron-задач, конфиг техжурнала и вспомогательные конфиги
(cron, logrotate). Здесь нет тестового фреймворка — вся проверка идёт через
синтаксис/статический анализ и точечный ручной прогон.

Репозиторий уже опубликован: `https://github.com/maxutovrg59/1c-maintenance`
(публичный, ветка `master`, remote `origin` настроен). Это влияет на пункт
"Публикация" ниже — речь идёт не о первой публикации, а о последующих пушах.

## Структура

```
1c-maintenance/
├── README.md                  # описание проекта, структура, установка — держать в актуальном виде
├── .gitignore                  # исключает rac.secret, .pgpass, logs/, *.dump, *.log
├── .gitattributes               # принудительный LF для *.sh/*.conf/*.xml и т.д.
├── etc/env.conf                # общий конфиг (пути, ИБ↔БД, retention) — плейсхолдеры, не секрет
├── etc/rac.secret.example       # шаблон логина/пароля администратора кластера rac
├── bin/common.sh                # общие функции log()/die()/require_cmd(), подключается через source
├── bin/*.sh                     # сами регламентные скрипты (см. README.md за списком)
├── tj_setup/logcfg.xml          # конфиг технологического журнала 1С
├── cron/1c-maintenance.cron     # расписание
└── logrotate/1c-maintenance.logrotate
```

Каждый `bin/*.sh` начинается с `source "$(dirname "${BASH_SOURCE[0]}")/common.sh"`,
затем задаёт `LOG_FILE`, использует `log INFO/WARN/ERROR "..."` и `die "..."` для
фатальных ошибок. Новые скрипты должны следовать этому же паттерну — не заводить
свою систему логирования.

## Понять структуру перед правками

Перед любым изменением сначала прочитать `README.md` этого проекта — там актуальные
списки скриптов, требования и расписание. Если правка меняет состав файлов,
параметры `env.conf` или расписание — README нужно обновить в том же коммите/изменении.

## Запуск и проверка изменений

Постоянно работающего сервиса нет, поэтому "запуск" — это:

1. Синтаксическая проверка всех скриптов: `bash -n bin/*.sh`.
2. Если доступен `shellcheck` — прогнать `shellcheck bin/*.sh` и поправить
   предупреждения (за исключением заведомо ожидаемых, например SC2154 для
   переменных из `env.conf`, которые подключаются через `source`, а не объявлены
   в самом файле).
3. Точечный ручной прогон конкретного скрипта только если есть тестовый/некритичный
   контур PostgreSQL — никогда не гонять `restore_test.sh` или `pg_maintenance.sh`
   на боевой базе без явного согласия пользователя (создание/удаление БД,
   `VACUUM` на боевых таблицах).
4. Для XML (`tj_setup/logcfg.xml`) — минимум проверка валидности XML
   (`xmllint --noout tj_setup/logcfg.xml`, если доступен).

## Внесение изменений

- Не хардкодить пути/имена ИБ/БД в скриптах — все специфичные для сервера значения
  идут через `etc/env.conf`.
- Не хранить реальные секреты (`etc/rac.secret`, реальный `.pgpass`) в репозитории —
  только `*.example`-шаблоны (проверять по `.gitignore`, что реальные файлы не отслеживаются).
- Придерживаться уже принятых соглашений: `set -uo pipefail` (без `-e`, т.к. скрипты
  сознательно продолжают обработку остальных ИБ при ошибке по одной из них и сами
  трекают `status`), логирование через `log()`, явные коды возврата.
- Любые доработки, которые предполагают исполнение на боевом сервере (реальный
  запуск `rac`, `pg_dump`, `VACUUM`, `DROP DATABASE` и т.п.) — только после
  явного подтверждения пользователя, это не readonly-анализ.

## Проверка ошибок перед коммитом

- `bash -n` по всем изменённым `*.sh`.
- Проверить, что во всех новых/изменённых файлах нет реальных паролей, хостов,
  IP или иных боевых данных — только плейсхолдеры и `.example`-шаблоны (репозиторий публичный).
- Проверить, что `README.md` синхронизирован со структурой (новые файлы отражены
  в дереве и таблице расписания, если расписание менялось).
- `git status`/`git diff` перед `git add`, чтобы не закоммитить что-то лишнее.

## Публикация изменений на GitHub

Репозиторий уже создан и опубликован, `origin` настроен — речь про обычный рабочий цикл:

1. Закоммитить изменения с понятным сообщением (что и почему, не просто "что").
2. **`git push` — только после явного подтверждения пользователя**, даже если
   изменение кажется мелким. Репозиторий публичный, push виден сразу всем.
3. Если меняется видимость репозитория, имя, или требуется что-то вроде
   force-push/перезаписи истории — это тоже отдельное явное подтверждение,
   такие операции не подразумеваются обычной просьбой "закоммить" или "обнови".
